# Claude Code モデル戦略

## TL;DR

**修正点が少ない・1〜2回の難判断**: Sonnetメイン + Opusサブを部分起動 → コスト約23%削減
**修正点が多い・難判断が連続**: 全Opusで進めた方が結局安価（サブ起動オーバーヘッド累積で逆転）

「節約のためにモデル切り替え」が逆効果になる閾値あり。**Opusサブを5回以上呼ぶならメインOpus一択**。

## 3パターンの実測比較（2026-05-02 Wiki監修セッション）

タスク: denken-hoki-wiki.html の実装済み6ページを3ペルソナ視点で監修

| パターン | 構成 | 重み付け消費 | A比 |
|---|---|---|---|
| A: Opusメイン全部 | Opus 1人で6ページ × 3視点審査 | 約485k相当 | 100% |
| B: Sonnetメイン + Opusサブ部分 | Sonnet主 + Opusサブで難判断のみ | 約375k相当 | **77%** |
| C: 全Sonnet（実際採用） | メインOpus + Sonnetサブ3並列 | 約325k相当 | 67% |

→ Bが理論上のスイートスポット。ただし「修正がとても多い」フェーズではこの優位性が崩れる。

## なぜ「修正が多い」と全Opusの方が安価になるか

サブエージェント起動には**1回あたり3-5kトークンのオーバーヘッド**が発生する：
- ペルソナ・基準ファイルのRead（毎回）
- プロンプトに必要文脈を全部詰め直し（毎回）
- 結果の要約・統合（毎回）

判断箇所が10個あると `10 × 4k = 40k` の純オーバーヘッド。Sonnetメインで節約した分を食いつぶす。

**逆転点の目安**: Opusサブ呼び出しが**5回を超えると**全Opusの方が結局安価。

## モデル選択の判断軸

| シーン | 最適 |
|---|---|
| 設計・PlanMode（1-2回） | Sonnet主 + Opus副 |
| Wiki監修・コードレビュー | Sonnet主 + Opus副（並列Sonnetサブで監修、最終承認だけOpus） |
| 大量Edit・テンプレ展開（5+箇所） | 全Sonnet |
| 修正フェーズ（連続的判断15箇所+） | **全Opus**（オーバーヘッド回避） |
| インタラクティブな設計議論 | 全Opus（コンテキスト連続性） |

## 落とし穴

### 1. 「すべてのモデル」メーターと「Sonnetのみ」メーターの誤読
- 「Sonnetのみ」枠は**メインがSonnetのとき**しかカウントされない
- Opusメイン内でSonnetサブを呼んでも「Sonnetのみ」枠は0%のまま
- → サブエージェントSonnetは「すべてのモデル」プールに合算

### 2. 部分Opus起動のサブは会話履歴を見ない
- メインの文脈を引き継がないので、プロンプトに必要情報を全部詰める必要あり
- CLAUDE.mdへの参照を含めるなど、毎回フル装備
- 詰め忘れると判断品質が落ちる

### 3. 「節約意識で頻繁にモデル切り替え」が逆効果
- 切替判断自体に思考コストがかかる
- セッション全体のテンポが落ちる
- **作業の性質が決まったら割り切って固定**するのが結局最速・最安

## 実践メモ

```bash
# パターンA: Sonnetメインで開始（修正点少なめ・難判断1-2回）
/model sonnet
# 必要時のみ Agent({model: "opus", ...}) で部分起動

# パターンB: Opusメインで開始（修正点多い・連続判断）
/model opus
# サブは並列Sonnet推奨（隔離された定型作業）
```

## 個人的判断軸（古舘ルール 2026-05-02）

- 「修正点が多い」と直感したら **迷わず全Opus**
- セッション中の「Opus呼び戻し回数」が3を超えたら、次は全Opusで再開
- コスト最適化に脳のリソースを割くより、**判断品質に集中**したほうが時間あたりの成果が大きい
