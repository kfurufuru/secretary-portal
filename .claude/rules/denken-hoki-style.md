#### 2.1.2 全体像SVGの必須化（v4.2/v2.5で確立）

表だけでは「条文間の方向性（上位→本条→委任）」が直感的に伝わらない。section 1 の① 全体像には**ハブ＆スポーク型のSVG概念図**を**表の直前**に必ず配置する：

```
        ┌──────────────────────┐
        │   省令第N条（上位）   │  赤系
        └──────────┬───────────┘
                   ↓ 上位
                   │
┌──────────────┐ ┌─▼──────────┐ ┌──────────────┐
│  並立/例外   │ │  本条（中央）│ │ 具体化/例外  │
│  （橙系）    │→│  （青系）    │←│ （橙〜青系） │
└──────────────┘ └─┬──────────┘ └──────────────┘
                   ↓ 委任
        ┌──────────▼───────────┐
        │  解釈第M条（委任先） │  緑系
        └──────────────────────┘
```

**SVGデザイン規範**：
- viewBox: 720×360（推奨）
- 中央: 本条（青系 `#e3f2fd` + `#1565c0`、太枠 stroke-width=2.5）
- 上方: 上位条文（赤系 `#ffebee` + `#c62828`）
- 左右: 並立/例外/具体化（橙系 `#fff3e0` + `#ef6c00`、青系 `#e1f5fe` + `#0288d1`）
- 下方: 委任先（緑系 `#e8f5e9` + `#2e7d32`）
- 矢印: 関係性ラベル（上位/並立/委任/具体化/例外）を明記
- 1行収納（CLAUDE.md「SVGは`<div>`で囲み空白行なし」ルール準拠）

**確立記事**: 56.md v4.3（実装型B）／5.md v2.6（規範型A）。

#### 2.1.3 全体像SVGと表の責務分離（重複ではなく相補）

「SVGがあれば表はいらない？」という疑問への回答：**両方必須**（教育心理学的根拠：Paivio の二重符号化理論）。

| 要素 | 役割 | 強み | 弱み |
|---|---|---|---|
| **SVG概念図** | 関係性の方向（上位／並立／委任）を視覚化 | 一目で構造把握、初学者の先行オーガナイザーに最適 | 詳細記述不可、スクリーンリーダー非対応、検索ヒットしない |
| **表** | 各条文の役割を文字で詳述 | 詳細記述可能、アクセシブル、grep/検索可能 | 関係性の方向が伝わりにくい |

**重複ではなく相補**: SVGは「**どう繋がっているか**」、表は「**何があるか・なぜそうか**」を伝える。両方読むことで視覚＋言語の二重符号化が起こり、記憶定着が単独表現の約2倍になる（教育心理学）。

**運用ルール**：
- SVGと表で**同じ情報を重複させない**（SVG=方向性のみ、表=詳細記述）
- SVG内のテキストは「条文番号＋一行要約」に絞る（詳細は表へ）
- 表のヘッダーは「条文／役割／第N条との関係」で統一
- 表の末尾に **→ 第N条は「[一行で核心]」の総括規定** という summary 行を追加

**SVG矢印デザインの注意**：
- すべての `<line>` には適切な `marker-end="url(#arrXXX)"` を必ず付ける（先端のない線は「未完成」に見える）
- marker は方向別に4種類定義（上向き arrTop / 右向き arrR / 左向き arrL / 下向き arrBot）
- 線の起点・終点はボックス境界にぴったり合わせる（5px以上の隙間は「浮いて見える」）

## 3. 新セクション「4. 法規特有表現」の必須テンプレート

各条文記事に section 4 として配置。中身は4ブロック構成（Dは56.md v3.0で新規確立）：

### ブロック A: 法規特有表現の早見表（5項目フォーマット、3〜5語）

```markdown
## 4. 法規特有表現

> 条文を読む前に、日常語と法規表現の対応を頭に入れておくと、原文の意味が一気に通る。

| 日常語 | 法規表現 | 試験での意味 | 混同注意 | 例文（条文調） |
|---|---|---|---|---|
| 大地と切り離す | 大地から絶縁する | 地絡電流の通路を断つ常時状態 | 「遮断」(動作)・「隔離」(空間)とは別物 | 電路は大地から絶縁しなければならない |
| 高低圧の電路がぶつかる | 混触 | 異なる電圧系統の物理的接触 | 「短絡」(同一系統内の接触)とは別 | 混触による高電圧の侵入等の異常 |
| アースを取る | 接地を施す | 大地と電気的に接続する | 「設置する」「接続する」では不可 | 保安上必要な接地その他の措置を講ずる |
```

### ブロック B: この条文で覚えるべき言い回し（条文の主題に応じて3〜5個を選択）

```markdown
### この条文で覚えるべき言い回し

- 「**施設する**」（設備を設けることを表す法規動詞）
- 「**保安上必要な措置を講ずる**」（例外条件の典型表現）
- 「**この限りでない**」（=「無条件に不要」ではなく「条件付き例外」）
- 「**危険のおそれがない**」（例外適用の前提条件）
- 「**通常の使用状態**」（試験範囲を限定する語）
```

### ブロック C: ひっかけ表現（混同パターン）

```markdown
### ひっかけ表現

| 誤った理解 | 正しい理解 |
|---|---|
| 「この限りでない」=無条件に不要 | 条件付き例外（危険なし or 保安措置あり） |
| 「電線」「配線」「電線路」「電路」をすべて同じ意味で使う | 別物（電路=通電部分、電線路=支持物含む一式 等） |

| 「アースを取る」と「接地を施す」を同じに扱う | 法規では「接地を施す」 |
```

## 4. 全条文共通の優先記載TOP10

### 7.1.1 最低3ソースクロスチェックの必須化（2026-05-02追加）

**1ソースの確認だけで数値を採用してはならない**。404・古い改正版・誤転記の可能性があるため、以下のルールを厳守：

| ステップ | 内容 |
|---|---|
| **S1: 一次ソース取得** | e-Gov / METI公式PDF を WebFetch で取得（最低1件） |
| **S2: 二次ソース取得（最低2件）** | 信頼性高二次資料を WebFetch で取得（404・タイムアウトなら別URL） |
| **S3: 数値の突合** | S1とS2両方の数値が一致するか確認 |
| **S4-A: 一致** | 採用。出典は一次ソース（METI PDF等）を明記 |
| **S4-B: 不一致** | 一次ソースを優先。なければ追加で二次ソースを探す（最低3ソース一致まで） |
| **S5: 並列WebFetch** | S1〜S2 は並列実行で時間短縮（合計2分以内） |

**禁止事項**：
- 1ソースだけで数値を採用すること
- 1ソースが404でクロスチェック不可能なまま記述すること
- ソースが古い改正版（例：平成20年版）のまま記述すること（最新版の確認を怠らない）

**過去事故**（§7.5）：56.md v3.0 で1ソース確認のまま公表 → 11A 誤記。複数サイトクロスチェックなら検出できた事故。

### 7.2 数値・条文番号の必須出典記載ルール

許容電流・絶縁抵抗・遮断時間・電圧区分・接地抵抗 などの**法令数値**を記載する場合：

1. **出典セルを必ず付ける**: 表の下に `出典：電技解釈第○条 別表第○（YYYY-MM-DD取得）` を明記
2. **公式ソース未確認なら数値を書かない**: WebFetchで公式ソース未照合の数値はテーブルに含めない（[要確認]フラグでの誤魔化しは禁止）
3. **「概念理解用」「目安」表記の禁止**: 法令数値に「概念用」「目安」と書いて誤魔化さない（実際の試験で誤答を誘発するため）
4. **単位と条件の明記**: 数値には必ず単位（A・MΩ・秒・kV）と条件（周囲温度30℃・絶縁種別・電線本数）を併記する

### 7.6 数値検証日記載ルール（2026-05-02 制定）

数値テーブルを含む記事には、**ページ末尾に「数値検証完了日」を明記**する。

**記載場所**: ページ末尾の `*最終確認: ... | ステータス: ... | ...*` 行

**記載フォーマット**:
```
*最終確認: YYYY-MM-DD | 数値検証完了: YYYY-MM-DD（PASS / CONDITIONAL PASS / INCONCLUSIVE） | ステータス: ... | [バージョニング基準](...)*
```

**判定区分**:
- **PASS**: 全数値が公式PDF（電技解釈・電技省令の最新版）と完全一致、出典セル付与済
- **CONDITIONAL PASS**: 公式照合できたが構造改善や軽微修正が残る
- **INCONCLUSIVE**: PDF読取不能等で公式照合未達。次回再評価が必要

**監修ログとの併記**:
ページ末尾の `## 監修ログ` セクションに以下を追加：
```
- **YYYY-MM-DD（数値検証）**: [監修者名]が[公式ソース]と照合 → [結果]。[修正内容]。**数値検証判定: [PASS/CONDITIONAL PASS/INCONCLUSIVE]**
```

**ルールの根拠**: 56.md / 58.md / 14.md 事故では「いつ・誰が・何を照合したか」が追跡できなかった。検証日を明示することで、記事の信頼度を読者・将来の自分が即判定できるようにする。

**運用**:
- 新規記事公開時：必ず数値検証完了日を記載
- 既存記事に数値修正があった場合：その日付で更新
- 電技解釈・電技省令の改正があった場合：再検証を実施し日付更新

新規事故が発生した場合、本表に追記する。事故は隠さず記録し、再発防止策を §7.3〜7.4 に反映する。

## 9. 条文タイプ判定フロー

### 9.1 判定の3軸

| 軸 | 規範型A | 実装型B | 構造型C |
|---|---|---|---|
| 条文の性格 | 原則・例外を定める | 具体仕様を定める | 用語を定義する |
| 数値の有無 | なし（解釈に委任） | あり | なし |
| 例 | 5・10・12・57条 | 14・22・56・58条 | 1条 |
| 合計セクション数 | 14 | 16 | 10〜11 |

### 9.2 判定手順（二択フロー）

```
START: 条文を読む
    ↓
Q1: 条文に具体的な数値・施設要件があるか？
    ├─ Yes → 【実装型B】（B型変奏を選ぶ）
    └─ No  → Q2
            ↓
Q2: 用語の定義が中心か？
    ├─ Yes → 【構造型C】（C型変奏を選ぶ）
    └─ No  → 【規範型A】（A型変奏を選ぶ）
```

### 9.3 混合型条文の扱い

条文によっては混合型がある（例：第56条は規範＋実装の混合だが、実装要件が中核）。
**迷ったら「数値・施設要件の有無」を最優先判定軸とする**。

混合型の判定基準:
- 数値・施設要件が **記事の中核（5割以上のボリューム）** → 実装型
- 原則・例外が中核で、数値は1〜2項目 → 規範型
- 用語定義と数値が混在（例：第1条に「定格電圧」の数値あり） → 構造型

### 9.4 タイプ判定の信頼度

CLAUDE.md行動原則 #9 に準拠：
- ≥90%（5・56条のように明確） → そのまま実装
- 70〜89%（短い条文・混合型） → 代替案を先に提示
- <70% → ユーザーに確認

### 9.5 判定結果の記載場所

各記事の冒頭メタ表「性格」行に明記：

```markdown
| 項目 | 内容 |
|------|------|
| 条文 | 電気設備技術基準 第○条 |
| 規定内容 | ... |
| 性格 | 原則規定（規範型） / 実装規定（実装型） / 用語定義（構造型） |
| 委任先 | 電技解釈第○条 |
```

これにより、後から検証スクリプトで `grep 性格` するだけでタイプを抽出できる。

## 9.7 デザイン規約（強制ルール・2026-05-03 制定）

DENKEN-WIKI監修部 デザイン班（アイブ・ラムス・原研哉・佐藤可士和・ショガー）が 56.md レビューで合意した恒久ルール。新規記事作成・既存記事の改訂で必ず遵守する。

### R1: admonition は4種類のみ使用

`!!!` および `???` の admonition タイプを以下4種に限定する。

| 採用 | 用途 |
|---|---|
| `note` | 補足情報・出題傾向・条文確認・例文取り扱い等の中立的情報 |
| `tip` | 暗記の核心・暗記のコツ・具体的な計算例（学習促進） |
| `warning` | 誤解防止・絶対禁止規定・誤った理解の指摘 |
| `success` | 穴埋め解答・正解選択肢（`???` の折りたたみ専用） |

**禁止**: `abstract` / `info` / `example` / `danger` / `failure` / `bug` / `question` 等。

**例外**: `??? question` は「セルフチェック」用に許容（折りたたみ問答パラダイムが note/tip と異なるため）。

**変換ルール**: 既存記事の `abstract` `info` → `note`、`example` → `tip` に置換する。

### R2: ==マーカー== は「穴埋め問題」専用、過去問頻出は **太字**

`==text==`（HTML出力 `<mark>`）は**穴埋め過去問関連のみ**、それ以外の強調は `**text**`。

| 用途 | マーカー | bold |
|---|---|---|
| **穴埋め空欄placeholder**（例：`==（ア）==`） | ✅ `==text==` | × |
| **穴埋め解答値**（`??? success` 内の正答セル：`==A==` `==10Ω==`） | ✅ `==text==` | × |
| **過去問で問われた箇所**（論説・計算問題で出題された語句） | × | ✅ `**text**` |
| 表の重要行・キーワード強調 | × | ✅ `**text**` |
| 条文引用の重要箇所 | × | ✅ `**text**` |
| メタ表の項目強調 | × | ✅ `**text**` |

**理由**: マーカー黄色は注意喚起の最強シグナル。穴埋め以外で使うと「強調インフレーション」を起こし、本当に重要な箇所（穴埋め解答）が埋もれる（Schoger原則4: hierarchy は色と太さで作る）。

**横展開時の自動変換スクリプト**（穴埋めplaceholderと success ブロック内は保持）:

```python
import re
def convert(text):
    SUCCESS_RE = re.compile(r'(\?\?\? (?:success|note)\s+"(?:解答|正答|答え)[^"]*"\n(?:    .*\n|\n)*?)(?=\n(?:[^ \n]|\?\?\?|\!\!\!)|\Z)', re.MULTILINE)
    blocks = []
    text = SUCCESS_RE.sub(lambda m: (blocks.append(m.group(1)), f'\n§§B{len(blocks)-1}§§\n')[1], text)
    PH_RE = re.compile(r'==（[ア-ヲ]+）==')
    phs = PH_RE.findall(text)
    for i, p in enumerate(phs): text = text.replace(p, f'§§P{i}§§', 1)
    text = re.sub(r'==(.+?)==', r'**\1**', text)
    for i, p in enumerate(phs): text = text.replace(f'§§P{i}§§', p)
    for i, b in enumerate(blocks): text = text.replace(f'\n§§B{i}§§\n', b)
    return text
```

**確立日**: 2026-05-03 — denken-wiki 全テーマ・解釈条文に横展開（commit `d6f67aa`）。344箇所の==マーカー乱用を整理

### R3: inline `<style>` ブロック禁止

`.md` 内に `<style>...</style>` ブロックを直接書かない。

**理由**: 同パターンが複数記事に重複展開されると、1箇所修正で全体反映できない（Long-lasting原則違反）。

**ルール**:
- 共通スタイルは `denken-wiki/docs/stylesheets/custom.css` に集約する
- design-tokens.css のトークン（`var(--space-4)` `var(--shadow-sm)` 等）を活用する
- 個別記事固有のスタイルは禁止（共通化できないなら設計を見直す）

**過去事例**: 56.md / 10.md / 15.md にあった `.article-overview-cross` 系統の inline `<style>` を 2026-05-03 に custom.css へ統合。

### R4: 装飾絵文字の制限

絵文字は**意味のある場合のみ**使う。

| 用途 | 採用例 | 禁止例 |
|---|---|---|
| 優先度マーカー | 🔴🟡🟢（section 12 ひっかけ） | — |
| 状態表示 | ✅ ❌ ⚠️ 🔥（試験頻度） | — |
| 索引アイコン | ⚡ 🔌 📜 等（全体像クロス図のノードに限る） | — |
| 装飾的な見出し prefix | × | `**📜 条文の核心**` のような前置絵文字 |
| メタ表の二重表記 | × | `🔥🔥 ★★☆☆☆`（同情報の二重） |

**変換ルール**: 装飾絵文字は削除し、テキストラベルだけにする。意味があるアイコンは維持する。

### R5: 情報重複の禁止（佐藤可士和原則）

- section 1 全体像表 と section 14 委任フロー表が**同じ情報を異なる視点で重複**してはいけない
- 重複が見つかったら、片方を「もう片方を参照」の引用ブロックに圧縮する
