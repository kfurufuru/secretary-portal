# ナレッジ管理ルール

## 基本思想

- L1（ログ）から昇格したもののみ記事化。思いつきはinboxへ
- 横展スタイル必須: 思考プロセスを残す。結論だけ書かない
- 情報の接続を記録: 「AとBがつながった瞬間」を価値とする
- 数字・具体例必須: 形容詞禁止。数値・事例で代替
- 抽象論禁止: 「重要です」「検討が必要です」で終わらない

## frontmatterスキーマ

```yaml
title: "日本語タイトル"
category: "電気計装 | 電験3種 | AI活用 | 設備管理 | 組織・マネジメント"
level: "draft | review | published"
created: "YYYY-MM-DD"
last_reviewed: "YYYY-MM-DD"
understanding_score: 3  # 1-5
source: "昇格元の記録"
tags: ["タグ1", "タグ2"]
related: ["knowledge/関連記事.md"]
```

## カテゴリ定義

| カテゴリ | 内容 | 代表タグ |
|---------|------|---------|
| 電気計装 | 保護リレー・接地・配線・計装ループ・規格解釈 | 保護協調, 接地, 計装 |
| 電験3種 | 電力・機械・理論・法規の学習ノート・過去問解析 | 理論, 電力, 機械, 法規 |
| AI活用 | プロンプト設計・ツール選定・業務自動化 | Claude, プロンプト, 自動化 |
| 設備管理 | 設備投資仕様・トラブル分析・保全手順 | 設備投資, 保全, 仕様検討 |
| 組織・マネジメント | 育成計画・評価・コミュニケーション | 育成, 評価, チーム |

## 理解度スコア

| スコア | 定義 | 電験3種での目安 |
|-------|------|--------------|
| 1 | 概念を読んだだけ | 用語は知っている |
| 2 | 大意はわかる。細部で詰まる | パターンはわかる |
| 3 | 自分の言葉で説明できる | 計算手順を再現できる |
| 4 | 応用問題に対応できる | 見慣れない問題でも解ける |
| 5 | 他者に教えられる | 誤答なし・理由まで説明 |

> スコア2以下はフェインマンセッション実施後に昇格

## L2昇格の判定基準

**基本ルール**: L2昇格は「自動ではなく、人間による理解確認」が前提。RSS フィード（ai-news/feeds/）からの推薦は参考に過ぎない。

### 昇格の必要条件
1. **理解度スコア ≥ 3**（自分の言葉で説明できる）
2. **実務適用性がある**（単なる情報ではなく、行動に結びつく）
3. **フェインマンセッション実施済み**（スコア2以下→3以上への昇格前に必須）

### 昇格候補の発見プロセス

| ステップ | 役割 | 備考 |
|---------|------|------|
| 1. feeds スキャン | link_generator.py | 同一 ai_category + score ≥3 の関連記事を自動抽出 |
| 2. weekly-digest 集計 | Dataview | 週単位で目立った記事を表示（参考情報） |
| 3. **人間による確認** | ユーザー | 「本当に理解しているか」をセルフチェック |
| 4. L2 昇格判定 | ユーザー | level: draft → review → published に手動更新 |

### 重要な区別

- **Dataview の役割**: 「昇格候補のリスト表示」（提案）
- **週次レビュー（weekly-digest.md）**: 「参考情報の集計」
- **L2 昇格**: 「理解度とスコアに基づく確実な判定」（自動ではない）

**反例**：score = 4 でも「読んだだけ」なら L2 ではなく、一度 inbox で保留してから昇格判定

## Do
- 結論を最初に書く（TL;DR必須）
- 思考プロセスを残す
- 数字で語る（抵抗値・電圧・条文番号・日付）
- 落とし穴を1件以上記録する
- 実践メモで現場に接続する
- 関連記事を横断接続する

## Don't
- 「重要です」で終わらない
- 抽象的な見出し（「概要」「まとめ」だけ）を立てない
- sourceなしにdraftを飛ばしてpublishedにしない
- 理解度スコア1で記事化しない
