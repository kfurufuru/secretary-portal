# 意思決定フレームワーク

## TL;DR

判断に迷ったときの評価軸を定義する。ひろゆきROI軸（実務最適化）と落合陽一創造軸（複雑系・横展性）の2軸で構造化する。

---

## 基本原則

- **結論先**：判断の迷いは評価軸の欠如が原因。軸を先に定義してから判断する
- **ROI定量化**：「良さそう」で進めない。実装コスト・効果・リスクを数値化する
- **ドメイン適合確認**：技術的に可能でも、対象ドメインと合わない投資はROIゼロ
- **盲点3点チェック**：軽量化の罠・手動運用累積・属人化リスクを必ず確認する

---

## 評価軸一覧

### 軸1: ひろゆきROI評価軸（実務最適化）

実装が本当に業務効率を改善するかを問う。感情的な「面白そう」を排除する。

| チェック項目 | 問い |
|-------------|------|
| 実装コスト | 何時間・何ステップかかるか？ |
| 効果 | 何が何分短縮・何件削減されるか？ |
| 手動運用コスト | 運用しつづけるコストは増えないか？ |
| 属人化リスク | 自分以外が引き継げるか？ |
| 代替案 | より低コストで同じ効果を得る手段はないか？ |

### 軸2: 落合陽一創造設計軸（複雑系・横展性）

知識・システムを可視化・接続することで生まれるインサイトと横展性を評価する。

| チェック項目 | 問い |
|-------------|------|
| ドメイン適合 | この可視化は対象ドメインと意味的に一致しているか？ |
| 横展性 | 他のドメインや文脈へ転用できるか？ |
| 認知効果 | 実装後に「気づき」が増えるか？数値化できるか？ |
| 複雑性コスト | 系の複雑化で保守性が下がらないか？ |

## Phase3 意思決定パターン

> ソース: `inbox/handoff-2026-04-23-Phase3-Decision.md`（2026-04-23 Opus検証）

### 判断対象: Python カタログ Phase3（3層マップ導入）

#### ひろゆきROI評価の適用

- **A案（3層マップ）の判断**: 不採用
  - L1/L2/L3 はナレッジ管理タクソノミー。Pythonツールカタログへの適用はドメイン不一致
  - ROIほぼゼロ：可視化コスト＞得られる効率改善
  - 代替判断: 知識体系の可視化が必要なら `portal-v2.html` 側で別途検討
- **Phase2**: 定義なし・不要のためスキップ確定
- **Phase1a**: 矢印フロー拡張のみ採用（実効性確認済み）

#### 落合陽一創造設計軸の適用

- 3層マップはドメイン不一致のため横展性ゼロと判定
- 「知識体系の可視化が必要なら別文脈（portal-v2.html）で検討」= ドメイン分離の原則を適用

#### 抽出した再利用可能パターン

- **パターン1: ドメイン適合チェックを最初に実施する**
  - 実装可能かどうかではなく、「そのドメインで意味を持つか」を先に確認する
  - 失敗例: L1/L2/L3タクソノミーをPythonツールカタログに適用 → 意味なし
- **パターン2: ROIゼロ判定は早めに切る**
  - 「面白い」「できる」はROIの根拠にならない
  - 実装コスト vs. 具体的な効率改善を比較し、改善がゼロなら即不採用
- **パターン3: スコープ外の価値は別文脈に移す**
  - 不採用案の価値が完全にゼロではない場合は「別ファイル・別フェーズで検討」として記録
  - 今回: 知識体系可視化 → `portal-v2.html` 側タスクとして温存

#### 発見したバグとの接続

Phase1a 実装検証中に構文エラー・CSS・SVG座標系の4件のバグを発見・修正済み。
意思決定の実行（Phase判断確定）がバグ発見のトリガーになった = **判断を先行させることで実装品質が上がる**。

## 盲点チェックリスト（全判断共通）

```
[ ] 軽量化の罠: 簡易実装に見えて、運用コストが隠れていないか？
[ ] 手動運用累積: 自動化しないと、毎回の手作業が積み重なっていないか？
[ ] 属人化: 自分以外が引き継げない設計になっていないか？
```
