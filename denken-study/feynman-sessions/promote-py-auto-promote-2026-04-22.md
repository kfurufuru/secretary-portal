# フェインマンセッション: promote.py 自動昇格機能

## 平易な説明（12歳向け）

### promote.py って何をするツールか

ノートの「下書き」「レビュー待ち」「完成」の3段階を自動で進めてくれるプログラム。  
図書館の本が「新刊コーナー → 普通の棚 → 特集コーナー」と格上げされるイメージ。

### 「どうやって格上げするか」の基準が2軸ある

1. **参照された回数（refs）**: 他のノートから「このノートを読んでね」と指されている数。図書館で他の本から「参考文献」に挙げられている数に相当する。
2. **アクセス日数**: 実際に「このノートを開いた」と記録した日の数。日記に「今日読んだ」と書いた回数。

### 格上げのしきい値（条件）

| 昇格ルート | 条件（どちらか一方でOK） |
|-----------|----------------------|
| 下書き → レビュー | refs≥2 **または** 30日以内アクセス≥3日 |
| レビュー → 完成 | refs≥5 **または** 累計アクセス≥7日 |

「どちらかOK」なのがポイント。refs が少なくてもよく読んでいれば格上げ、逆も然り。

### 実際に使うコマンド

```bash
# アクセスを手動記録
py promote.py --log-access knowledge/some-note.md

# 自動昇格を実行（詳細ログあり）
py promote.py --auto-promote --verbose

# アクセス統計を確認
py promote.py --access-stats
```

### オマケ機能: クロスリンク自動提案

`suggest_related_files` という機能がついていて、「このノートはあのノートと関係ありそう」と自動で教えてくれる。これを使ってリンクをどんどん貼ると、refs カウントが自然に増えやすくなる。

## 詰まったポイント（知識の穴）

### 穴①: `--log-access` は「手動」記録であることの意味

最初に読んだとき「アクセスログって自動で取れるんじゃないか」と思った。  
しかし実装は **手動記録** 方式。これはなぜか、どういうトレードオフか、が即答できなかった。

### 穴②: refs カウントの計測ロジック

「クロスリファレンス数を数える」と書いてあるが、具体的に **どのファイル内のどの記述を数えるのか** がわからなかった。Markdownの `[[wikilink]]` か？ 相対パスの `[text](./path)` か？ 両方か？

### 穴③: draft→review と review→published でしきい値が非対称な理由

| ステップ | refs | アクセス日数 |
|---------|------|-------------|
| draft→review | ≥2 | ≥3日（30日以内） |
| review→published | ≥5 | ≥7日（累計） |

なぜ後者は「30日以内」という期間制限がなく「累計」なのか。設計意図がわからなかった。

## 穴を埋める解説

### 穴①の解説: 手動記録を選んだ理由

**自動計測できない制約がある。** promote.py はローカルのMarkdownファイルを管理するCLIであり、ファイルアクセスイベントをOS側でフックする仕組みを持っていない。Webサーバーのアクセスログとは異なり、テキストエディタでファイルを開いても promote.py には通知が来ない。

このため「実際に読んで活用した」という事実を本人が `--log-access` で能動的に記録する設計になっている。

**トレードオフ**: 記録漏れがあると過小評価になる（実際よりアクセス少なく見える）。対策は週次レビューのルーティンに記録を組み込むこと。

### 穴②の解説: refs カウントの計測対象（コード確証済み）

`count_cross_references` 関数（promote.py 383-407行）の実装を確認した。

**対象ディレクトリ**（386行）:
```python
search_dirs = [KNOWLEDGE_DIR, INBOX_DIR, SECRETARY_DIR / "digital-twin"]
```
`knowledge/`, `inbox/`, `digital-twin/` の3ディレクトリ全 `.md` ファイルを走査。

**マッチするリンク形式**（389-393行、4パターン）:
```python
patterns = [
    re.compile(rf"\[\[{re.escape(stem)}\]\]", re.IGNORECASE),      # [[note-name]]
    re.compile(rf"\[\[{re.escape(filename)}\]\]", re.IGNORECASE),   # [[note-name.md]]
    re.compile(rf"\({re.escape(filename)}\)", re.IGNORECASE),       # (note-name.md)
    re.compile(rf"knowledge/{re.escape(filename)}", re.IGNORECASE), # knowledge/note-name.md
]
```
Obsidian `[[wikilink]]` 形式（stem・フルファイル名の両方）とMarkdown `(path)` 形式とパス記述の計4パターン。

**カウント単位**（396-407行）: ファイル単位（1ファイルに複数リンクがあっても count+1 は1回のみ）。自己参照は除外（400行）。

**結論**: refs カウントは「何ファイルから参照されているか（ファイル数）」であり、1ファイル内の複数リンクは1としてカウントされる。リンクを張らずに参照したケースはカウントされない点は前回の推論どおり。

### 穴③の解説: draft→review と review→published のアクセス条件（コード確証・前回推論修正）

**実装確認**（promote.py 517行・537行）:

```python
# draft チェック（519-533行）
access = count_access_days(rel)          # days=30 デフォルト
acc_ok = access >= DRAFT_TO_REVIEW_ACCESS  # ≥3

# review チェック（535-549行）
access = count_access_days(rel)          # days=30 デフォルト（同じ引数）
acc_ok = access >= REVIEW_TO_PUBLISHED_ACCESS  # ≥7
```

**前回の推論誤り**: フェインマンセッション当初「review→publishedは累計カウント」と推論していたが、コードを確認すると両方とも `count_access_days(rel)` をデフォルト引数（days=30）で呼び出している。つまり **両方とも「30日以内のユニーク日数」**。

**実際の差はしきい値のみ**:
| ステップ | 期間 | 必要アクセス日数 |
|---------|------|---------------|
| draft→review | 直近30日以内 | ≥3日 |
| review→published | 直近30日以内 | ≥7日 |

「累計」という概念は実装にない。「30日以内に7日以上アクセスされた」という高頻度条件が published の昇格基準。設計意図は「1ヶ月内にほぼ毎週参照される程度の実用性がある知識」を published とする判断基準と解釈できる。
