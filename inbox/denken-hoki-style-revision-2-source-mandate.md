## 改訂の経緯（事故サマリ）

電技省令56条記事 v3.0で許容電流値を完全に誤記した。

| 項目 | 誤記値 | 正しい値 |
|---|---|---|
| 1.6mm 銅線（VVF）許容電流 | 11A | **27A** |
| 電流減少係数（3条以下） | 「約90%」 | **0.70** |

**原因の3層分解**:
1. 並列エージェントへ法令数値の生成を「概念理解用」と説明したため、エージェントが既存記事の不正確な値を引き継いで書き出した
2. ユーザー（私）が `[要確認]` フラグを付けて誤魔化す形になり、公式照合が後回しになった
3. v3.0レビュー（47/50 S評価）でも数値の照合は行われず、構造評価のみで合格判定した

**ユーザーからの指摘**:「サイトが使い物にならない」「再発防止を講じて」

→ §7を「公式ソース必須化」へ全面改訂し、並列エージェントへの絶対ルール R1〜R5 を厳格化する。

```markdown
## 7. 信頼度・記載ルール（公式ソース必須化）

### 7.1 公式ソースの定義

電気法令の数値・条文番号・出典は、以下のいずれかの**公式ソース**から取得する：

| ソース | URL | 用途 |
|---|---|---|
| e-Gov法令検索 | https://laws.e-gov.go.jp/law/409M50000400052/ | 電技省令の原文 |
| METI 電技解釈PDF | https://www.meti.go.jp/policy/safety_security/industrial_safety/law/files/dengikaishaku.pdf | 電技解釈の原文・別表 |
| 電気技術者試験センター | https://www.shiken.or.jp/ | 過去問問題文 |
| 電験王3 | https://denken-ou.com/ | 過去問解説（信頼度高い二次資料） |

二次資料（書籍・他Wiki・YouTube解説）は**参考用**であり、数値の根拠としては使わない。

### 7.2 数値・条文番号の必須出典記載ルール

許容電流・絶縁抵抗・遮断時間・電圧区分・接地抵抗 などの**法令数値**を記載する場合：

1. **出典セルを必ず付ける**: 表の下に `出典：電技解釈第○条 別表第○（YYYY-MM-DD取得）` を明記
2. **公式ソース未確認なら [要確認] フラグ**: WebFetchで公式ソース未照合の数値には必ず付ける。ただし [要確認] フラグは「次セッションで必ず公式照合する」TODOであり、放置禁止
3. **「概念理解用」「目安」表記の禁止**: 法令数値に「概念用」「目安」と書いて誤魔化さない（実際の試験で誤答を誘発するため）
4. **単位と条件の明記**: 数値には必ず単位（A・MΩ・秒・kV）と条件（周囲温度30℃・絶縁種別・電線本数）を併記する

### 7.3 並列エージェントへの法令数値生成タスクの絶対ルール

複数のセッションで起きた事故（56.md v3.0）を踏まえ、以下を**絶対ルール化**する。
違反時はユーザーが該当セッションを即停止し、エージェント生成物を破棄する。

| ルール | 内容 | 違反時の影響 |
|---|---|---|
| **R1: 公式ソース未参照なら数値を書かない** | エージェントは記憶や推測で数値を書かず、必ず WebFetch で公式PDFを参照する | 誤値がサイトに掲載される（56.md v3.0事故） |
| **R2: 出典セット禁止** | 「許容電流値（[要確認]）」のような出典なしテーブルを作らない。出典なしなら表自体を作らない | 誤値が見過ごされる |
| **R3: プロンプトに公式URL同梱必須** | 並列エージェント呼び出し時、公式URLとWebFetchテンプレを必ず渡す | エージェントが推測で書く |
| **R4: 「概念理解用」エクスキューズ禁止** | 「実装は要確認」と書いて誤値を載せない。概念解説のみで数値テーブルを書かない | サイトが使い物にならなくなる |
| **R5: ピアレビュー必須** | 数値テーブルを書いたら別エージェントでソース照合させてから本反映 | 単独レビューで見過ごす |

**R3で同梱するプロンプトテンプレ**:

```
あなたは電気法令記事の section 7（数値実装表）を書きます。
以下の手順を厳守してください：

1. 必ず WebFetch で次のURLを取得すること：
   https://www.meti.go.jp/policy/safety_security/industrial_safety/law/files/dengikaishaku.pdf
2. 取得したPDF内の「別表第○」から数値を直接抽出すること
3. 記憶・推測・既存記事からの引き継ぎは絶対に禁止
4. 抽出できなかった数値は、表に書かず「公式PDF未抽出のため省略」と明記
5. 表の下に必ず出典行を入れること：
   「出典：電技解釈第○条 別表第○（YYYY-MM-DD取得）」
6. 数値の単位（A, MΩ, 秒, kV）と条件（周囲温度・絶縁種別等）を併記すること

違反した場合、生成物は破棄されます。
```

### 7.4 §6章（責務分離）における「数値実装表」の必須化条件

実装型B型の section 7「数値実装表」を書く前提：

- [ ] 公式ソース（METI PDF など）を WebFetch で取得済み
- [ ] 抽出した数値を表に転記済み
- [ ] 出典行（`出典：電技解釈第○条 別表第○`）を表の下に追加済み
- [ ] 数値の「単位」「条件」（周囲温度・絶縁種別等）を明記済み
- [ ] 並列エージェント生成なら、ピアレビューで別エージェントが照合済み

→ 上記5項目を**ひとつでも満たさない場合、section 7 自体を書かない**（旧版の「11A の誤値」のような事故を再発させない）。
section 7 を省略した場合は、ユーザー本人が手動で公式PDFから数値を転記する（エージェント任せにしない）。

## 既存§7の維持点と削除点

### 完全置換（既存§7 4項目はすべて削除）

既存§7は CLAUDE.md行動原則 #9 を引用しただけの簡素な4項目だった：

1. 「条文・eGov公式未確認の数値・条文番号は `[要確認]` フラグ付き」
2. 「ブロックAの例文は条文原文を貼らず、原文に倣った要約にする」
3. 「例文が条文そのままになる場合は、!!! info ブロックで…」
4. 「並列エージェントには…未確認の数値・条文番号は記載しない指示」

→ いずれも**抽象的で守られなかった**（特に1と4）。新§7は「公式URLを WebFetch で取得すること」「出典セルを必ず付けること」「ピアレビュー必須」という**手順レベル**まで具体化する。

### 維持する原則

- 条文原文の複製禁止 → 新§7.1の二次資料扱いと整合（条文原文は e-Gov リンクのみ案内）
- 並列エージェントへの指示明示 → 新§7.3 R3 でプロンプトテンプレ化
- 複製不可な部分は `!!! info` ブロックで「e-Gov公式から転記」プレースホルダー → 新§7.4 の「section 7 を書かない」と整合

## 想定される反論と回答

| Q | A |
|---|---|
| 常に公式PDF取得していたら遅くなるのでは？ | 単発取得で30秒、ピアレビュー含めても2分。「サイトが使い物にならない」となるコスト（記事全削除・読者の信用喪失）の方が遥かに大きい |
| 概念理解だけの数値表は作っても良いのでは？ | ダメ。学習者は「ここに書かれている=正しい」と信じて使う。誤値を「概念用」と書く誤魔化しは試験で誤答を誘発する |
| WebFetchがタイムアウトしたら？ | section 7 自体を書かない（§7.4）。後日ユーザー本人が手動で転記する |
| 既存記事（v1.0〜v3.0）の数値は全部チェックするのか？ | 次フェーズで `denken_law_audit.py` を作成して一括照合する。本草案の適用範囲は新規記事から |
| ピアレビューのエージェントも誤るのでは？ | §7.3 R5のピアレビューは「公式PDF再取得 → 数値突合」を行う。同じプロンプトテンプレ（§7.3 R3）を渡すため、独立検証になる |
