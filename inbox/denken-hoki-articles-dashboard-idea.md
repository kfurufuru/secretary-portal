# 設計メモ: 法規Wiki articles/ ダッシュボード化（C案）

## 背景

2026-05-01 セッションで A案（`hide: [toc]` で右TOC非表示）を即実装。
ホリエモン社員から「中央が単なるリンクの山。情報密度が低い」と指摘あり。
将来 C案として中央を **学習ダッシュボード** に作り変える設計メモ。

## 課題

現状の `articles/index.md`：
- 全78条文への箇条書きリンクが並ぶだけ
- どの条文が**頻出**か、どこが**自分の弱点**かが分からない
- ジョブズ案D（ページ廃止）は却下したが、現状では存在意義が薄い

## 設計方針

中央ページを「**今日何を学ぶべきか即判断できる学習ハブ**」に変える。
左サイド = 全条文への直リンク（変更なし）。中央 = 意思決定支援。

## コンポーネント案

### 1. 法令カード（4枚）

```
┌──────────────────────────────────┐
│ 🟥 電気事業法                       │
│ 16条文 / 過去問頻出: ★★★             │
│ 最近5年出題: 12問                    │
│ あなたの正答率: 67% (8/12)           │
│ 弱点条文: 第38条, 第44条             │
│ → 詳細一覧                          │
└──────────────────────────────────┘
```

4枚×法令（事業法・省令・解釈・その他）。クリックで該当法令トップへ。

### 2. 出題頻度ヒートマップ

各条文を「過去5年の出題回数」で色分け（白→濃い緑）。
Notion 過去問DB（551件）から集計。

### 3. 弱点TOP5

`denken3-study-dashboard` の正答率データから「あなたが間違えた条文TOP5」を抽出。
クリックで該当条文ページへ直行。

### 4. 学習進捗バー

```
電気事業法: ████████░░ 8/16 既学習
省令:       ██████░░░░ 12/21 既学習
解釈:       █████░░░░░ 18/38 既学習
```

### 5. 「今日のおすすめ」

頻出 × 未学習 × 弱点 の3軸スコアで1問選定。

## 技術構成

### データソース

| データ | 場所 | 取得方法 |
|--------|------|---------|
| 過去問出題実績 | Notion DB ff4fcb73cc14408caedf87c904ae2fd9 | update_dashboard.py（既存） |
| 正答率 | denken3-study-dashboard/data/records.json | 既存 |
| 条文メタ（重要度・難易度） | mkdocs.yml の extra: または別yml | 新規作成必要 |

### 実装パターン

MkDocs Material は静的サイトなので、選択肢：

**Option α: ビルド時注入（推奨）**
- macros プラグイン（既導入）でMarkdownにJinjaテンプレで埋め込む
- メリット: 静的サイトで完結、JSランタイム不要
- デメリット: ビルド時点のスナップショット（リアルタイム性なし）

**Option β: クライアントサイドJS**
- HTML +static JSON で動的描画
- メリット: 履歴更新で即反映
- デメリット: MkDocs Material の枠組み外、メンテ負担

→ **Option α 採用**。週次の `update_dashboard.py` 実行時に articles/index.md も自動再生成。

## 実装ステップ（1〜2時間想定）

1. **データ取得スクリプト**: update_dashboard.py を拡張、法令別集計データを `_data/articles_stats.yml` に出力（30分）
2. **macros 関数追加**: `articles_stats(houto)` を main.py に実装、Jinjaから呼べるように（20分）
3. **articles/index.md 改修**: 法令カード4枚・ヒートマップ・弱点TOP5・進捗バー・今日のおすすめ をJinjaで記述（40分）
4. **CSS追加**: カードレイアウト・ヒートマップ色（10分）
5. **検証・調整**: ローカル mkdocs serve → ハードリフレッシュ確認（10分）

## 落とし穴予測

1. **過去問データのカバー範囲**: 法規551件中、条文紐付けが不完全な可能性 → 紐付けスクリプトを別途作る必要あるかも
2. **「あなたの正答率」の個人化**: 静的サイトでは個人データ表示が困難 → クライアント JS で localStorage から読むか、静的に「最新スナップショット」を埋め込むか判断必要
3. **ヒートマップ200条文の描画コスト**: SVG または CSS Grid で軽量実装
