## 6. §9（新規）「条文タイプ判定フロー」完全版

```markdown
## 9. 条文タイプ判定フロー

### 判定の3軸

| 軸 | 規範型 | 実装型 | 構造型 |
|---|---|---|---|
| 条文の性格 | 原則・例外を定める | 具体仕様を定める | 用語を定義する |
| 数値の有無 | なし（解釈に委任） | あり | なし |
| 例 | 5・10・12・57条 | 14・22・56・58条 | 1条 |
| コア9 + 変奏 | A型変奏（5セクション） | B型変奏（5セクション） | C型変奏（2セクション） |
| 合計セクション数 | 14 | 14 | 11 |

### 判定手順（二択フロー）

```
START: 条文を読む
    ↓
Q1: 条文に具体的な数値・施設要件があるか？
    ├─ Yes → 【実装型】（B型変奏を選ぶ）
    └─ No  → Q2
            ↓
Q2: 用語の定義が中心か？
    ├─ Yes → 【構造型】（C型変奏を選ぶ）
    └─ No  → 【規範型】（A型変奏を選ぶ）
```

### 混合型条文の扱い

条文によっては混合型がある（例：第56条は規範＋実装の混合だが、実装要件が中核）。
**迷ったら「数値・施設要件の有無」を最優先判定軸とする**。

混合型の判定基準:
- 数値・施設要件が **記事の中核（5割以上のボリューム）** → 実装型
- 原則・例外が中核で、数値は1〜2項目 → 規範型
- 用語定義と数値が混在（例：第1条に「定格電圧」の数値あり） → 構造型

### タイプ判定の信頼度

CLAUDE.md行動原則 #9 に準拠：
- ≥90%（5・56条のように明確） → そのまま実装
- 70〜89%（短い条文・混合型） → 代替案を先に提示
- <70% → ユーザーに確認

### 判定結果の記載場所

各記事の冒頭メタ表「性格」行に明記：

```markdown
| 項目 | 内容 |
|------|------|
| 条文 | 電気設備技術基準 第○条 |
| 規定内容 | ... |
| 性格 | 原則規定（規範型） / 実装規定（実装型） / 用語定義（構造型） |
| 委任先 | 電技解釈第○条 |
```

これにより、後から検証スクリプトで `grep 性格` するだけでタイプを抽出できる。
```

## 8. 想定される反論と回答

### Q1. 「タイプ判定が曖昧で迷う条文があるのでは？」

**A**. §9の二択フローに加え、3つの判定基準を明示した：
- 数値・施設要件が中核（5割以上） → 実装型
- 原則・例外が中核 → 規範型
- 用語定義が中心 → 構造型

迷う場合は信頼度<70%として **ユーザー確認** に回す。属人化を防ぐ。

### Q2. 「3変奏で割り切れない条文（混合型）はどうする？」

**A**. 第56条が典型的な混合型（規範＋実装）。判定軸は「中核ボリューム」で、数値・施設要件が中核なら実装型を優先する。
混合型は概念的にはありえても、**変奏は1つを選ぶ**ことで運用を単純化する（§9に明記）。

逆に「規範型変奏A＋実装型変奏Bを混在させる」のは禁止。テンプレ複雑化の温床になる。

### Q3. 「テンプレ複雑化で運用コストが上がるのでは？」

**A**. むしろコスト低下が見込める：

- **コア9セクションは全タイプ共通** → 共通テンプレを使い回せる
- **変奏は5 or 2セクション** → 選択肢が3つに限定され、判定迷いが減る
- **検証フローは4段階＋型ごとのセクション数チェック** → 自動化容易
- **既存記事5.md・56.mdは無変更** → 即時の修正コストはゼロ

逆に **「14セクション一律強制」のままだと**、56条の実装型強み（数値表・設計フロー）が削られて品質低下を招く。56条リライト前の状態に戻ることになる。

### Q4. 「変奏セクションの番号がタイプによって変わる（A型は7・8、C型は省略）と読者が混乱しないか？」

**A**. 全タイプで「**1〜5はコア前半、6〜10は変奏、11〜13はコア後半**」という大枠を統一する。
読者にはメタ表「性格」でタイプが告知されるため、変奏内容の差は予期できる。
