# Claude Code で電験 Wiki 100記事を1日で監査・修正した話 — 並列エージェント56本 + AI社員3名で運用するメソッド

> **TL;DR**
>
> - **2システム計約132記事** を **2日** で監査 → 公開停止級6件・要修正13件・主題ミスマッチ多数を検出
> - 並列エージェント **累計60本** を **6→6→10→7→8→4** の波で投入（fan-out + 集約 + fan-out の入れ子）
> - **AI社員 3名**（落合陽一 = 構造 / ひろゆき = ROI / ホリエモン = 選択と集中）に **2-1 で割れたら少数派を尊重** という運用ルールを敷いた
> - **METI WebFetch は本番で全失敗** したので、最初にローカル PDF キャッシュを確保するのがクリティカル
> - 学んだことを **6本の feedback memory** に落とし、次回は自走できるようにした

ここから先は、Claude Code を「個人ナレッジサイトの誤記殲滅」に振り切って運用した実例の話です。読者対象は、Claude Code か Claude Agent SDK を業務で使い始めて「単発の補助以上にスケールさせたい」と感じている人。

---

## 1. 監査対象と最終成果

| 項目 | denken-wiki（条文逐条解説） | hoki-wiki（法規暗記Hub） |
|---|---|---|
| 形式 | MkDocs Material（Markdown 100記事） | React SPA（JSX 32ページ） |
| 監査前の状態 | v1.0 〜 v2.7 が混在 | 4-5月の急増ページ |
| 検出 | 公開停止級 6件 / 要修正 13件 / 構造問題 5件 | 致命 4件 / 重要 11件 / 軽微多数 |
| 修正コミット | 5本（Phase A → B → C → D-A → D-B） | 2本（Phase Hoki-A → B） |

**「誤記の種類」が両システムでほぼ同じだった** ことが最大の発見でした。代表例：

- **架空の問題番号**（電験三種法規は問1〜13までしか存在しないのに「問14」が登場）
- **条文番号の取り違え**（§227 と §232 が別主題なのに混同）
- **改正未追従**（小出力発電設備 → 小規模発電設備 への 2022 改正が反映されていない）
- **典拠不明の暗記式**（試験では問題文が式を与えるのに「公式」として記載）

つまり **執筆プロセス自体に共通の弱点パターン** があり、Wiki が違ってもエージェント設計は流用できる、という仮説が立ちました。

## 2. AI 社員 3名構成

社内会議のメタファです。3名の Subagent をペルソナとして固定し、設計判断のたびに発言させます。

| ペルソナ | 担当視点 | 発言例 |
|---|---|---|
| 落合陽一 | 構造的負債・将来設計 | 「条文番号を本文に直書きしている時点で、改正のたびに横展開コストが線形で増える」 |
| ひろゆき | ROI・コスト | 「30 記事の全面書き直しは 5 セッション。今やる必要ある？」 |
| ホリエモン | 選択と集中・外販 | 「監査ノウハウは資産になる。修正してる暇あったら記事化しろ」 |

### 2-1 多数決の罠と判定ルール

最初は「2-1 で多数派採用」していましたが、これが **構造的負債領域で痛い目に遭った**。具体的には：

> ひろゆき＋ホリエモン「軽量化のため警告 admonition だけで済ます（パッチ路線）」
> 落合「いま直さないと旧 v1.0 由来の誤記が新規ページにコピペされ続けるぞ（構造路線）」
>
> → 多数決で **パッチ路線採用** → 翌セッションで新規ページに同じ誤記が再現

このあと「**少数派が構造的負債を指摘していたら、4 トリガー条件を満たすときは少数派を尊重する**」というルールを feedback memory に記録しました。トリガー条件は、

1. 少数派の指摘が「将来コスト」軸
2. 多数派が「短期工数」軸
3. 同じ問題の再発履歴がある
4. 横展開対象が 3件以上ある

の AND 条件です。ここまで構造化すると、人間が不在のときも Claude 自身がルールを呼び出して判断できます。

## 3. 並列エージェント設計：fan-out の入れ子

エージェントを「いつ並列で投げて、いつ直列で投げるか」が肝でした。基本パターンは：

```
[監査フェーズ]
  Phase 1 機械検査（リンク・記号・空要素・原文照合）           1 agent
  Phase 2 一次ソース照合（kijun / kaishaku / jigyoho / other） 4 agent 並列
  Phase 3 過去問照合（kakomon.yml）                            1 agent
                                                              ─────
                                                              6 agent

[修正フェーズ]
  Phase A 公開停止級 6件 修正                                  6 agent 並列（1記事1agent）
  Phase B 要修正 13件                                          10 agent 並列
  Phase C 後発監査 + warning 配置（44記事）                    7 agent 並列
  Phase D-A kakomon.yml ベース 10件                            6 agent 並列
  Phase D-B 経産省PDF ベース 7件 + 旧条番号体系再構築          8 agent 並列
                                                              ──────
                                                              累計 56+ agent
```

ポイントは、

- **監査は完全並列**（読み取りのみで競合しない）
- **修正は「1ファイル＝1エージェント」で並列化**（Edit 競合を避けるため、同じ md を複数 agent が触らない）
- **集約フェーズは直列**（fan-in は親 Claude 1人で実施。SUMMARY-phaseC.md 等の集約 md にまとめる）

React SPA 側（hoki-pages.jsx 1ファイルに 32ページ）は同一ファイル競合があるため、**並列監査 → 直列修正** に切り替えました。これは Edit ツールの直列性に依存した設計判断です。

## 4. METI WebFetch 全失敗 → ローカルキャッシュ先行戦略

電技解釈は **eGov 法令API に登録されておらず**、経産省告示PDF だけが一次ソース。これを WebFetch で取りに行こうとして全敗しました。

> 結論：**並列監査の前に PDF を `scripts/cache/` に手動配置するのが必須**

これを守らないとどうなるかは Phase C で実証済みで、44記事の監査が **「wiki 内クロスリファレンスのみ」に退行** し、誤記同士の循環照合（誤りが誤りを正しいと証明する）が発生しました。具体的には：

- kaishaku/68.md「§68 = 低圧の高さ規定／§75 = 高圧の高さ規定で +1m」
- kaishaku/65.md「高さは §68（低圧）／§75（高圧）が正典」

の **2記事がお互いを参照して整合性 PASS** していました。実際には §75 は「電車線等との接近・交差」の離隔距離規定で、§68 が低高圧共通の高さ規定。Phase Hoki 横展開監査で隣の Wiki から見たとき初めて矛盾が顕在化しました。

**横展開監査が逆方向に正典確定の触媒になる**、というのが今回の二次発見です。

## 5. 検出された誤記の分類

100 + 32 記事から出た誤記は、たった 5 つの failure mode に集約できました：

| failure mode | 例 | 検出方法 |
|---|---|---|
| 1. 架空情報の創作 | 「免状の有効期限3年」（現行法に存在しない） | eGov 法令API 照合 |
| 2. 条文番号取り違え | §227 ↔ §232、§38 ↔ §227 | 経産省PDF 照合 |
| 3. 改正未追従 | 「小出力発電設備」（2022改正前用語） | KAISEI-YYYY-NNN ラベルで版管理 |
| 4. 主題ミスマッチ | ページ名「使用前自主検査」だが中身は月次点検 | 過去問 kakomon.yml topic 突合 |
| 5. wiki 内循環参照 | §68 と §65 が互いを誤って証明 | **隣接Wiki から見直す** |

このうち 1〜3 は **一次ソースキャッシュ** があれば機械的に検出できます。4〜5 は人間判断（あるいは AI 社員議論）が要ります。

## 6. 再現可能なフロー（90分で誰でも始められる）

```
[準備フェーズ・15分]
1. リポジトリ全文を Glob/Grep で棚卸し
2. 一次ソース PDF を scripts/cache/ に手動配置
3. kakomon.yml 相当の出題実績DBを用意（無ければまず作る）

[監査フェーズ・並列45分]
4. 機械検査エージェント 1名（リンク・記号・空要素）
5. 一次ソース照合エージェント N名（カテゴリ別 4-6並列）
6. 過去問照合エージェント 1名
7. 親 Claude が SUMMARY.md に集約

[判断フェーズ・人間15分]
8. AI 社員 3名で「修正 vs パッチ vs 据置」を議論
9. 2-1 で割れたら 4トリガーで少数派採用判定

[修正フェーズ・並列15分]
10. 1ファイル＝1エージェントで Edit を並列発射
11. wiki_check.py で構文チェック
12. ビルド → push → 50秒待ち → DOM 検証

[学習フェーズ・5分]
13. 今回の判断・誤りを feedback memory に1ファイル/件で保存
```

90 分で 1 サイクル回ります。エラー検出率は導入直後で 1セッションあたり 10-20件です。

## 8. まとめ

- **個人ナレッジサイトでも誤記は必ず溜まる**。Wiki の規模が 50 記事を超えたら定期監査が必要。
- **Claude Code + 並列エージェント + AI 社員** の組み合わせは、定額月額で再現可能なメソッドになっている。
- **一次ソースキャッシュ先行**・**横展開監査**・**少数派尊重ルール** が再発防止の核心。
- 学んだ判断は **1件1ファイルの feedback memory** に書き出して、次回 Claude が自走できる状態にする。

このメソッドを採用してから、誤記が見つかる速度が **手作業時代の 10 倍以上** になりました。逆に言えば、Wiki の「真の出題傾向」をデータベース化し始めた瞬間、執筆精度が AI 側で律速される時代になった、ということです。

## 付録：使った Claude Code 機能

- **Subagent (Task tool)**: 監査・修正・集約の全工程
- **Edit tool**: 1ファイル1エージェントで Edit 競合回避
- **Glob / Grep**: 100記事全文棚卸し
- **WebFetch**: 失敗。METI サイト相手は最初から PDF キャッシュ前提
- **feedback memory（CLAUDE.md 直下に1件1ファイル）**: 判断ログ・横展開ルール
